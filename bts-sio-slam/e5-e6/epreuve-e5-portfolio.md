# Le portfolio E5 : à quoi il sert et ce qu'il doit contenir

Le **portfolio** est le support de votre présentation à l'épreuve E5 du BTS SIO.
Il ne remplace pas le **tableau de synthèse**, mais il le complète.

Le tableau permet au jury d'identifier rapidement vos réalisations.
Le portfolio permet, lui, de **les comprendre** et de **les vérifier**.

Autrement dit :

- le **tableau de synthèse** résume
- le **portfolio** explique et apporte les preuves

Le portfolio doit donc montrer de manière claire :

- ce que vous avez réalisé
- dans quel contexte
- quel était votre rôle
- quelles compétences ont été mobilisées
- quelles preuves vous pouvez présenter

---

# Ce que le jury attend

L'épreuve E5 n'est pas une démonstration technique en direct.
Ce n'est pas non plus une soutenance de code.

Le jury attend surtout une **présentation professionnelle et structurée** de votre parcours et de vos réalisations.

Le portfolio doit donc aider le jury à répondre rapidement à ces questions :

- Qui est le candidat ?
- Quel est son parcours et a quoi se destine-t-il ?
- Dans quel cadre a-t-il travaillé ?
- Qu'a-t-il réellement fait ?
- Avec quels outils et quelles méthodes ?
- Quels résultats ont été obtenus ?
- Quelles compétences du référentiel ont été mobilisées ?
- Quelles preuves permettent de le vérifier ?

---

# Ce qu'il faut mettre dans le portfolio

## 1. Une page d'accueil simple

La première page doit être sobre et claire.

Elle peut contenir :

- nom et prénom
- spécialité : développeur fullstack, développeur front, etc.
- diplôme préparé : **BTS SIO**
- option : **SLAM**

Inutile de surcharger cette page.
Son rôle est simplement d'identifier le candidat.

---

## 2. Une présentation du candidat

Cette partie permet d'expliquer votre parcours.

On doit y retrouver :

- votre formation
- vos stages ou expériences
- vos centres d'intérêt professionnels
- votre projet de poursuite d'études ou votre projet professionnel

L'objectif n'est pas de raconter toute votre vie, mais de donner au jury un **fil conducteur**.

Exemples de questions auxquelles cette partie répond :

- Pourquoi avez-vous choisi le BTS SIO ?
- Pourquoi l'option SLAM ?
- Vers quels métiers ou études souhaitez-vous aller ?

Pensez au storytelling lors d'un entretien d'embauche : qui êtes vous et a quoi va servir ce diplôme, c'est l'étape vers quel objectif ? 
---

## 3. Une présentation de vos réalisations professionnelles

C'est le coeur du portfolio.

Chaque réalisation présente dans le **tableau de synthèse** doit pouvoir être retrouvée dans le portfolio.

Une réalisation peut être :

- un projet réalisé en formation
- une mission effectuée en stage
- une amélioration ou correction sur une application existante
- une installation ou configuration technique
- un déploiement
- une contribution significative à un projet collectif

Attention :
une simple liste de projets ne suffit pas.
Pour chaque réalisation, il faut expliquer ce que vous avez réellement fait.

---

# Structure conseillée pour chaque réalisation

Pour chaque projet ou mission, vous pouvez suivre la structure suivante.

## 1. Titre de la réalisation

Le titre doit être précis.

Exemples :

- Développement d'une application web de gestion de stock
- Création d'une API REST pour la gestion des utilisateurs
- Correction et évolution d'un site e-commerce
- Déploiement d'une application Symfony sur un serveur Linux

---

## 2. Contexte

Expliquez où et dans quel cadre la réalisation a été menée.

À indiquer :

- formation ou stage
- nom de l'entreprise si nécessaire
- besoin de départ
- situation ou problème à résoudre

Exemple :

> Pendant mon stage, l'entreprise avait besoin d'un outil permettant de centraliser les demandes d'intervention des utilisateurs.

---

## 3. Objectif ou besoin

Expliquez clairement à quoi devait servir la solution.

Exemples :

- automatiser une tâche
- faciliter la gestion des utilisateurs
- mettre à disposition un service web
- corriger un dysfonctionnement
- répondre à un besoin métier précis

Cette partie doit montrer que vous comprenez **pourquoi** le projet existait.

---

## 4. Votre rôle personnel

C'est un point essentiel.
Le jury veut savoir ce que **vous** avez fait, pas seulement ce que le projet faisait.

Vous devez donc préciser :

- vos tâches
- vos responsabilités
- ce que vous avez développé, configuré, corrigé ou déployé
- ce que vous avez fait seul ou en équipe

Exemples :

- conception de la base de données
- développement de l'interface utilisateur
- création d'API
- rédaction de documentation technique
- réalisation de tests
- déploiement sur le serveur

---

## 5. Outils et technologies utilisés

Indiquez les outils réellement utilisés pendant la réalisation.

Exemples :

- langages : PHP, JavaScript, SQL, Python
- frameworks : Symfony, React, Laravel, Express
- bases de données : MySQL, PostgreSQL
- outils : Git, Docker, Postman, Trello, Figma
- environnement : Linux, Apache, Nginx

L'idée n'est pas de faire une liste impressionnante, mais une liste **cohérente**.

---

## 6. Démarche ou étapes principales

Résumez les grandes étapes du travail.

Exemple :

- analyse du besoin
- conception de la base de données
- développement des fonctionnalités
- tests
- correction des anomalies
- mise en production

Cette partie permet au jury de comprendre comment vous avez mené le travail.

---

## 7. Résultat obtenu

Expliquez ce que la réalisation a produit concrètement.

Exemples :

- application fonctionnelle
- site mis en ligne
- bug corrigé
- service déployé
- gain de temps pour l'entreprise
- amélioration de l'expérience utilisateur

Le résultat doit être observable et compréhensible.

---

## 8. Compétences mobilisées

Faites le lien avec les compétences du référentiel E5.

Par exemple :

- gérer le patrimoine informatique
- répondre aux incidents et aux demandes d'assistance
- développer la présence en ligne de l'organisation
- travailler en mode projet
- mettre à disposition des utilisateurs un service informatique
- organiser son développement professionnel

Il ne s'agit pas de cocher toutes les compétences, mais de ne garder que celles qui sont réellement justifiées.

---

## 9. Preuves à intégrer

Le portfolio doit contenir ou référencer des preuves.

Exemples de preuves utiles :

- captures d'écran
- extraits de documentation
- schémas UML ou schémas d'architecture
- lien vers un dépôt Git
- lien vers l'application
- documentation utilisateur
- documentation technique
- compte-rendu de tests
- ticket, demande ou cahier des charges simplifié

Les preuves servent à montrer que la réalisation est réelle et que vous pourrez en parler à l'oral.

---

# Deux façons d'organiser le portfolio

## Option 1. Entrée par les projets

Vous créez une page ou une section par réalisation.

C'est souvent la structure la plus simple.
Elle permet au jury de naviguer facilement entre les projets.

Dans ce cas, chaque projet contient :

- contexte
- besoin
- rôle
- outils
- résultat
- compétences
- preuves

## Option 2. Entrée par les compétences

Vous organisez le portfolio selon les compétences du référentiel.

Exemple :

- Gérer le patrimoine informatique
- Répondre aux incidents
- Travailler en mode projet

Puis, dans chaque compétence, vous rattachez une ou plusieurs réalisations.

Cette approche est possible, mais elle est souvent plus difficile à construire.

En pratique, pour l'E5, l'organisation **par projets/réalisations** est généralement la plus lisible.

---

# La place de la veille technologique

Le portfolio doit aussi contenir une partie sur la **veille technologique**.

Cette partie doit montrer :

- le sujet suivi
- pourquoi ce sujet a été choisi
- les sources utilisées
- les outils de veille utilisés
- ce que cette veille vous a apporté

Exemples d'outils :

- Flux RSS
- newsletters
- YouTube
- LinkedIn
- GitHub
- sites spécialisés
- agrégateurs de contenus

Le plus important n'est pas de dire "je fais de la veille", mais d'expliquer **comment** vous la faites et **ce qu'elle vous apporte**.$

ATTENTION : vous ne devez pas dire que vous faites une recherche internet mais que l'information vient automatiquement a vous grace aux outils choisis !

---

# Ce qu'il faut éviter

Voici les erreurs les plus fréquentes.

## 1. Faire un portfolio trop vague

Exemple :

- "J'ai fait un site web"
- "J'ai développé une application"

Ce n'est pas suffisant.
Il faut préciser le contexte, le besoin, votre rôle et le résultat.

## 2. Décrire le projet sans expliquer votre travail personnel

Le jury évalue votre activité.
Il faut donc toujours distinguer :

- ce que faisait le projet
- ce que vous avez fait dans le projet

## 3. Mettre uniquement du code

Le portfolio n'est pas un dépôt de code source.
Il doit surtout être un support d'explication et de preuve.

## 4. Faire un portfolio trop chargé

Trop de texte décourage la lecture.
Trop peu de texte empêche de comprendre.

Il faut viser un bon équilibre :

- informations utiles
- formulation simple
- preuves bien visibles

## 5. Oublier le lien avec le tableau de synthèse

Si une réalisation apparaît dans le tableau, elle doit être retrouvée facilement dans le portfolio.

---

# Ce qu'on peut retenir

Un bon portfolio E5 doit permettre au jury de voir rapidement :

- qui vous êtes
- ce que vous avez réalisé
- dans quel contexte
- ce que vous avez personnellement fait
- avec quels outils
- quels résultats ont été obtenus
- quelles compétences ont été mobilisées
- quelles preuves sont disponibles

En une phrase :

> un portfolio E5 n'est pas une vitrine décorative, c'est un support clair de communication professionnelle et de justification de vos réalisations.
