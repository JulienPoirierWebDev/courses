Pour beaucoup d’étudiants de **BTS SIO**, le vrai blocage n’est pas de remplir le tableau… c’est **de trouver leurs réalisations professionnelles**.
Ils pensent souvent : *« Je n’ai rien fait d’intéressant »* alors qu’ils ont en réalité **10 à 20 réalisations exploitables**.

La solution efficace consiste à utiliser une **stratégie d’auto-questionnement structurée**, proche d’une **rétrospective de projet**.

Je te propose une méthode pédagogique que les étudiants peuvent suivre seuls.

---

# Stratégie d’auto-questionnement

## Identifier ses réalisations professionnelles (BTS SIO – E5)

Objectif :
Transformer **les activités réalisées pendant la formation et les stages** en **réalisations valorisables dans le tableau de synthèse**.

---

# Étape 1 — Reconstituer sa chronologie

Avant de chercher des réalisations, il faut **retrouver ce que vous avez réellement fait**.

Posez-vous ces questions :

* Quels **projets avons-nous réalisés en cours ?**
* Quels **TP importants avons-nous faits ?**
* Quels **outils ou technologies avons-nous appris ?**
* Qu’ai-je fait **pendant mon stage ?**
* Quelles **tâches m’ont été confiées ?**

Écrivez **tout ce qui vous vient**, même si cela paraît banal.

Exemple :

* création d’une API Symfony
* projet React
* base de données SQL
* installation Linux
* déploiement Docker
* correction de bugs
* création script Python
* gestion Git

À ce stade :

➡️ on ne sélectionne rien
➡️ on **collecte**

---

# Étape 2 — Identifier les actions réelles

Une réalisation doit contenir **une action technique identifiable**.

Posez-vous ces questions :

* Qu’est-ce que **j’ai créé ?**
* Qu’est-ce que **j’ai configuré ?**
* Qu’est-ce que **j’ai corrigé ?**
* Qu’est-ce que **j’ai amélioré ?**
* Qu’est-ce que **j’ai automatisé ?**

Exemples :

| Activité vague  | Réalisation identifiable                   |
| --------------- | ------------------------------------------ |
| Projet web      | Développement d’un site web avec React     |
| Base de données | Conception du schéma SQL d’une application |
| Linux           | Installation d’un serveur Debian           |
| Support         | Correction d’un bug dans une API           |

---

# Étape 3 — Vérifier la valeur professionnelle

Toutes les activités ne deviennent pas des réalisations.

Posez ces questions :

### 1️⃣ Y a-t-il un objectif ?

Exemple :

* permettre l’inscription à un tournoi
* gérer les utilisateurs
* stocker des données

---

### 2️⃣ Y a-t-il une solution technique ?

Exemple :

* API REST
* base de données
* script
* serveur

---

### 3️⃣ Y a-t-il un résultat ?

Exemple :

* application fonctionnelle
* service en ligne
* bug corrigé
* système opérationnel

---

Si les **trois existent**, c’est une réalisation.

---

# Étape 4 — Identifier les preuves

Une réalisation doit pouvoir être **prouvée**.

Posez ces questions :

* Ai-je du **code source** ?
* Ai-je des **captures d’écran** ?
* Ai-je une **documentation** ?
* Ai-je un **schéma** ?
* Ai-je un **dépôt Git** ?

Si oui :

➡️ la réalisation est exploitable.

---

# Étape 5 — Identifier les compétences mobilisées

Maintenant demandez-vous :

### Qu’est-ce que j’ai réellement fait ?

Exemples :

| Action               | Compétence                    |
| -------------------- | ----------------------------- |
| installer serveur    | gérer patrimoine informatique |
| corriger bug         | gérer incident                |
| créer site web       | développer présence en ligne  |
| utiliser Git         | travailler en mode projet     |
| déployer application | mettre à disposition service  |

---

# Étape 6 — Transformer en réalisation

Utiliser ce modèle simple :

```
Titre

Objectif

Technologies utilisées

Résultat

Documents disponibles
```

---

### Exemple

```
Développement d'une API REST de gestion de tournois

Objectif :
permettre la gestion des clubs et des tournois sportifs.

Technologies :
Node.js, Express, Sequelize, MySQL

Résultat :
API fonctionnelle permettant CRUD des clubs et tournois.

Documents :
- code GitHub
- documentation API
- captures Postman
```

---

# Étape 7 — Vérifier que la réalisation est pertinente

Une bonne réalisation doit répondre à ces questions :

* Est-ce **technique** ?
* Est-ce **compréhensible par un jury** ?
* Est-ce que **je peux l’expliquer à l’oral** ?

Si oui → garder.

---

# Méthode rapide : la règle des 5 questions

Pour identifier une réalisation, posez-vous :

1️⃣ Quel **problème** devait être résolu ?
2️⃣ Quelle **solution technique** ai-je mise en place ?
3️⃣ Quelles **technologies** ai-je utilisées ?
4️⃣ Quel **résultat** ai-je obtenu ?
5️⃣ Quelles **preuves** puis-je montrer ?

Si vous pouvez répondre aux 5 → c’est une réalisation.

---

# Exemple d’auto-questionnement réel

Étudiant :

> "J’ai fait un projet React."

Auto-questionnement :

* pourquoi ?
* pour faire quoi ?
* avec quelles données ?
* comment ?
* résultat ?

Transformation :

```
Développement d'une application React de gestion d'événements

Objectif :
permettre la création et la gestion d'événements sportifs.

Technologies :
React, API REST, MySQL

Résultat :
interface permettant l'inscription des participants.
```

---

# Nombre idéal de réalisations

Objectif réaliste :

6 à 10 réalisations.

Répartition typique :

| Origine   | Nombre |
| --------- | ------ |
| formation | 3 à 4  |
| stage 1   | 2 à 3  |
| stage 2   | 2 à 4  |

---

# Astuce pédagogique (très efficace)

Demander aux étudiants de compléter cette phrase :

> **"J’ai utilisé ___ pour résoudre ___ afin de ___."**

Exemple :

> J’ai utilisé **Node.js et Sequelize** pour résoudre **la gestion des clubs** afin de **créer une API REST**.

C’est souvent **la base d’une réalisation E5**.
